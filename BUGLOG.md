### P7 · Сеть: таймаут запросов (раздел 5)
- **Что уже было ОК:** `api-client.js` дружелюбно обрабатывает offline (reject fetch → `errors.network`), 5xx/cold-start (502/503/504 → `errors.server_busy`), non-JSON, 401/403/404/429; есть кольцевой буфер `_dar_api_errors` для диагностики. Холодный старт Vercel закрыт сообщением «сервер перегружен».
- **Симптом (пробел 5.1):** у `fetch` НЕ было таймаута. При зависшем соединении (частое в мобильном Telegram WebView — потеря сети на полуслове) запрос висел бесконечно → юзер перед вечным спиннером без обратной связи.
- **Сделано:** в `request()` добавлен `AbortController` с таймаутом 90с (щедрый — выше любой легитимной AI-генерации Oracle/послание/разбор, но ловит истинный hang). При abort → `kind:'timeout'` → новое дружелюбное сообщение `errors.timeout` (ru/en/es). Если `AbortController` недоступен — graceful degradation (работаем без таймаута). `finally clearTimeout` чтобы не течь.
- **Остаётся (отмечено, не критично):** `getOracleForRelative` делает прямой `fetch` мимо `request()` — без таймаута и friendlyError. Меньшая поверхность; не трогаю сейчас, чтобы не задеть поток Оракула для близких.
- **Инвариант:** 1105/1105/1106. Cache-bust 20260608c → 20260608d.
- **Статус:** ✅ Исправлено. Каркас качества разд. 5 (таймауты/сеть/холодный старт).

### Обновления «не доходили» до Telegram — нужна была ручная чистка кэша
- **Симптом:** после деплоя Светлана не видела свежий код в Telegram Mini App, сколько ни чистила кэш. Казалось «Vercel не деплоит».
- **Диагностика:** Vercel деплоит штатно — прод `yupdar.com` отдавал актуальный токен и заголовки `Cache-Control: no-cache, no-store, must-revalidate`, HTML свежий. Запушенный код = origin/main. Проблема НЕ в Vercel и НЕ в действиях Светы.
- **Причина:** Telegram WebView держит старый `index.html` в системном кэше, который не сбрасывается ни «очисткой кэша», ни сворачиванием. Раз сам HTML старый — внутри него старые `?v=` ссылки, и cache-busting токен не помогает (до него не доходит).
- **Сделано (самолечение версии):** добавлен `public/version.txt` + инлайн version-check в `<head>` `index.html`. При запуске грузит `/version.txt?_=<ts>` (cache:'no-store'); если серверная версия ≠ зашитому `BUILD` — один раз форсит `location.replace` на URL с уникальным `_v=<версия>`, и Telegram берёт свежий HTML. Анти-петля через `sessionStorage._vreload` (ровно 1 релоад на новую версию). Проверено в превью: версия совпадает → молчит; версия новее → 1 форс-релоад, дальше guard. Теперь обновления подтягиваются САМИ, без чистки кэша.
- **ВАЖНО на каждый деплой:** бампить ВМЕСТЕ `public/version.txt`, константу `BUILD` в `index.html` и токен `?v=` всех ассетов. Текущая версия `20260606f`.
- **Статус:** ✅ Исправлено 06.06.2026 (self-healing version check, версия 20260606f).

### Серверный AI-текст приходил на русском при EN/ES
- **Симптом:** даже при EN/ES по-русски оставались: послание Оракула («Сегодня твоя дочь Ангелина...»), «Полное послание», главы детской «Книги для родителей», контент «Сегодня» в АРКА.
- **Причина (несколько слоёв):**
  1. Префикс `applyLanguage` (EN/ES-инструкция) добавлялся сверху, но в ТЕЛЕ промптов оставались строки «Только русский / Никакой латиницы / Не пиши иностранных слов» — LLM мог послушаться их.
  2. `content.js` → `sanitizeOracleOutput`/`spellCheckOracleOutput` вырезали ВСЮ латиницу (`replace(/[a-zA-Z]+/g,'')`) и гнали русский Яндекс-спеллер → уничтожали английский/испанский текст.
  3. `message.js` → `russifyText` (строка 95) стирала любое латинское слово целиком + `feminizeText` — тоже фатально для EN/ES.
  4. Кэш Оракула (`oracle_cache`) и детской книги (`child_book_sections`) не учитывал язык → русский кэш за день перебивал EN/ES.
- **Решение Светы (06.06.2026):** правило «только русский» в промптах действует ТОЛЬКО когда выбран русский. При EN — весь ответ на английском, при ES — на испанском, без примеси других языков. Это снимает блокировку с замороженного промпта В РАМКАХ языка (тело промптов НЕ менялось — закон `law_message_prompt_final.md` соблюдён).
